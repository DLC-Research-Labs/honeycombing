# Ensemble Explainer Schema (Objective 3)

Date: 2026-07-03
Status: schema shipped; first real payload ingested — ALARM NC congressional ensemble (`nc-congress-2020-alarm`, status `draft`) via `scripts/build-alarm-ensemble.mjs`. The mock remains as a test fixture. Findings + calibration in `docs/research/outputs/alarm-ensemble/nc-alarm-ensemble-ingestion.md`.

## Purpose

Ensembles (thousands of simulated legal plans) answer "what would random compliant maps look like?" Honeycombing's job is to *explain* that answer spatially — project percentile and outlier positions onto districts and H3 cells — without weakening the statistical meaning of the ensemble itself. This schema is the contract between ensemble tools and the UI.

## Where the schema lives

- Types + helpers: `app/lib/honeycomb-ui-helpers.ts` (`EnsembleSummary`, `ENSEMBLE_IMPORT_SCHEMA`, `classifyEnsemblePercentile`, `getEnsembleOutlierGate`)
- Registry: `public/data/ensembles/registry.json`
- Mock payload: `public/data/ensembles/nc-congress-2020-mock.json`, generated by `scripts/build-mock-ensemble.mjs` (deterministic literals — regenerating is byte-stable)

## Shape

An `EnsembleSummary` has three layers:

1. **`method`** — generator, algorithm, plan count, seed, and a required `constraints` list (population deviation, contiguity, county splits, VRA handling). This is the comparison universe. It is not metadata; it is the meaning of every percentile downstream.
2. **`seatMeasure`** — the headline distribution (e.g. Democratic seats of 14) as a histogram, with `comparedPlans` positioning known registry plans (court, enacted) inside it by value and percentile.
3. **`unitMeasures[]`** — distributions keyed by `district`, `precinct`, or `h3` unit ids. District-keyed measures index by the `referencePlanId` plan's district ids; h3-keyed measures set `h3Resolution`. Each unit carries monotonic `p5/p25/p50/p75/p95` percentiles plus the compared plan's value and percentile.

All `planId`/`referencePlanId` values must resolve in the plan registry, so ensemble positions inherit known plan provenance.

## The outlier gate

`getEnsembleOutlierGate(summary)` blocks outlier display when:

- `status === "mock"` — fabricated numbers must never be cited;
- `method.constraints` is empty — no documented comparison universe, no outlier claim;
- `planCount <= 0`.

`classifyEnsemblePercentile` bands: `<5` low_outlier, `5–25` low_edge, `25–75` typical, `75–95` high_edge, `>95` high_outlier. UI must present the constraint list before or alongside any band coloring (Objective 3 success criterion), and outlier bands render only when the gate allows.

## Ingestion targets (industry-standard sources to stack)

| Source | What it provides | Adapter mapping |
| --- | --- | --- |
| **ALARM 50-State Redistricting Simulations** (Harvard Dataverse; `redist` SMC) | 5,000-plan ensembles per state with per-plan, per-district summary stats; published constraints per state | **INGESTED** (`build-alarm-ensemble.mjs`): `pre_20` per-district votes → seat histogram + rank-matched district `unitMeasures`; documented NC constraints → `method.constraints`; DOI → `method.citation`. Plan assignment matrices (`plans.rds`) not yet ingested — blocks the h3-keyed measure |
| **GerryChain / MGGG ReCom runs** | Self-generated ensembles with explicit updaters and constraints | Run outputs → same shape; our run config → `method` verbatim. Use when we need NC-specific constraint variants ALARM didn't publish |
| **mggg-states / Redistricting Data Hub shapefiles** | Precinct geographies + election results + demographics | Not an ensemble input — a substrate cross-check. We already carry VEST 2020; RDH/mggg-states lets us diff our precinct joins against theirs before any evidence claim |
| **MGGG published reports (e.g. NC study)** | Reviewed ensemble findings | Calibration only: our ingested distributions should be consistent with published medians before we trust an adapter |

Cell-level projection (the H3 explainer itself) requires plan assignment files (precinct→district per ensemble plan), which ALARM publishes; the adapter aggregates "share/distribution of the containing district's measure" per H3 cell — the mock's `h3_district_dem_share` measure models exactly this.

## What the mock is and is not

`nc-congress-2020-mock.json` exists so tests and UI can be built against the full shape (both band directions are exercised: mock enacted plan at the 0.5th percentile, mock court plan at 68.3). Every number is fabricated; `mockCaveat` says so, the gate enforces it, and the histogram intentionally resembles no published NC ensemble closely enough to be mistaken for one.
